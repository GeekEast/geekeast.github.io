Javascript 模块系统

背景

  • 数量众多的JavaScript文件在载入html时，会产生同名变量在全局冲突的现象。
  • 下面的两个writeHello会产生变量冲突的问题

解决方案1: 命名空间模式

  • 本质: 变量挂载在一个对象上面
  • 存在问题: 分层处理无法完成封装，数据依然是public的

解决方案2: IIFE+闭包

  • 根据闭包数据无法访问的特性，完成了数据封装的任务
  • 函数会形成自身的作用域，其中声明的变量不会污染到全局，这个任务由IIFE来完成
  • 闭包的产生必须有对象或者函数的返回，这个任务由IIFE来完成
  • 而且模块之间是可以通过传参来进行沟通的
  • 存在问题: 以上解决方案的模块加载都是运行时加载，因此依赖关系需要保证正确的加载顺序，才能得以实施。

解决方案3: XXXXJS

CommonJS
  • 运行在服务器端
  • 同步加载
  • 运行时加载:
  • 每个文件就是一个模块，有自己的作用域(函数作用域)，内置一个module对象
  • require实际上会编译和运行js文件中的所有的内容
  • 模块可以多次加载，但是只会在第一次加载时运行一次, 然后运行结果就被缓存了，以后再加载，就直接读取缓存结果。要想让模块再次运行，必须清除缓存。
  • 模块加载的顺序，按照其在代码中出现的顺序。
  • exports负责开发数据接口，模块文件中的this默认指向了module.exports
  • 复制值而不是复制引用
RequireJS
  • 用在浏览器端
  • 运行时加载
  • 异步加载
  • 基于AMD
  • 依赖前置
SeaJS
  • 用在浏览器端
  • 运行时加载
  • 异步记载
  • 基于CMD
  • 即时加载

解决方案4: Module Pattern

  • 编译时加载
  • 是浏览器端和服务器端的通用解决方案
  • 加载: import,只能写在顶层
  • 导出: export
  • 复制引用而非复制值

CommonJS vs AMD(CMD) vs ES Module

属性 CommonJS AMD(CMD) ES Module
加载时机 运行时加载 运行时加载 编译时加载
加载内容 对象 对象 接口
加载程度 全部载入 全部载入 部分载入
拷贝机制 拷贝值 拷贝值 拷贝引用
this指向 当前模块的exports对象 当前模块的exports对象 undefined

注意点:

  • ES Module 加载的不是对象，因此无法访问加载的模块本身
